# 6 模型推理篇

## 6.1 推理两阶段：Prefill和Decode

在基于 Transformer 的大模型推理中，整个生成过程可分为两大阶段：

预填充（Prefill）阶段 ： 对完整的输入prompt进行一次性并行计算 ，为每一个token生成并缓存键（Key）和值（Value）向量（即KV cache）。这一步只需运行一次，缓存内容将用于后续的生成步骤；

解码（Decode）阶段 ： 模型以自回归方式逐步生成新 token。 每一轮仅需计算当前要生成的 token，并与此前缓存的 KV （包括用户查询和已经生成的token）进行注意力计算，从而避免对整个历史序列重复计算，显著降低了计算量。

### 6.1.1 Prefill阶段

Token 化 ：输入为批大小为 batch_size 的若干prompt，每个 prompt 被处理为长度为 seq_len 的 Token 序列。

嵌入层 ：这些Token序列会被映射为隐藏向量 X ，其形状为 (batch_size, seq_len, hidden_dim) 。

线性映射 ：通过权重矩阵 【公式开始】W_Q, W_K, W_V 【公式结束】 （形状为 (hidden_dim，num_heads * head_dim) ），分别计算得到 【公式开始】Q, K, V 【公式结束】 ，并重塑为 (batch_size, num_heads, seq_len, head_dim) 。

KV cache缓存 ：计算得到的 K 和 V 矩阵会被缓存下来，供后续解码阶段使用。

注意力计算：

【公式开始】A = softmax(\frac{Q \cdot K^T}{\sqrt d}) 【公式结束】

其中 【公式开始】A 【公式结束】 的形状为 (batch_size, num_heads, seq_len，seq_len) ，计算复杂度为 【公式开始】O(batch\_{size} * num\_{heads} * seq\_{len}^2 * head\_{dim}) = O(batch\_{size} * seq\_{len}^2 * hidden\_dim) 【公式结束】 。

【公式开始】O = A \cdot V 【公式结束】

其中 【公式开始】O 【公式结束】 形状为 (batch_size, num_heads, seq_len, head_dim) ，随后会被重塑为 (batch_size, seq_len, hidden_dim) 并传递到下一层，计算复杂度同上。

这里只列出了复杂度最高的两步，总体计算复杂度为 【公式开始】O(batch\_{size} * seq\_{len}^2 * hidden\_dim) 【公式结束】 。

生成第一个token ：在工程实践中， 为减少一次显存读写和前向传播开销，通常会直接利用 prompt 最后一个位置的 hidden state来生成第一个token。

为什么要做prefill？

避免重复计算 ：每一步解码都依赖于之前所有生成的中间结果， 如果不在prefill阶段缓存 KV，每生成一个新 token，都需要对整个历史序列（prompt + 已生成的 tokens）重新跑一次完整的自注意力计算 ，计算复杂度为 O( 【公式开始】seq\_len^2 【公式结束】 )，效率很低。

提高GPU 利用率 ：Prefill阶段是典型的 计算密集型任务 ，需要进行大规模的矩阵乘法运算。由于输入prompt是完整的，可以 通过高度并行来最大化发挥 GPU 的算力优势 。

便于并行优化 ： 将prefill和decode阶段解耦后，可以分别在不同的GPU上执行 。 prefill 阶段适合 张量并行 来加速首token的生成，提升TTFT（Time To First Token），而decode 阶段则适合用 数据并行或流水线并行 来提升逐token的生成吞吐量，也就是TPOT（Tokens Per Output Time，生成两个连续生成的词元之间的平均时延）。 这种解耦方式可以针对不同阶段采用最合适的并行策略，既能缩短首token的响应时间，也能提升整体生成速度。（相关结论可参考论文 DistServe: Disaggregating Prefill and Decoding for Goodput-optimized Large Language Model Serving ）

### 6.1.2 Decode阶段

输入准备 ：使用prefill阶段已经计算并存储的 KV cache， 以及 当前轮刚生成的 token（第一次decode时，是prefill阶段生成的第一个token）。

嵌入层 ：在第t步，将新生成的 token 转换为嵌入向量，计算得到 【公式开始】Q_t, K_t, V_t 【公式结束】 ，并将 【公式开始】K_t, V_t 【公式结束】 更新到KV cache中。

计算注意力 ：基于 【公式开始】Q_t 【公式结束】 和现有的KV cache进行注意力计算，计算复杂度为 【公式开始】O(seq\_len) 【公式结束】 ，相比prefill阶段大幅降低。

生成下一个token ：将最后一层的注意力输出经过线性层和softmax，得到下一个token的概率分布，并根据解码策略（如greedy search、beam search）选出下一个token，作为下一次decode的输入。

Decode阶段是典型的 通信密集型 任务，主要受限于显存带宽。因为生成一条回复通常需要多次decode， 每次都需要访问和更新KV cache。随着生成内容长度的增加，KV cache也会越来越大，对GPU内存带宽的压力也随之增大 。

Python
# 基于KV cache实现的prefill和decode阶段
import torch
import torch.nn as nn
import torch.nn.functional as F

class MultiHeadAttentionWithKVCache(nn.Module):
 def __init__(self, hidden_dim, num_heads):
 super().__init__()
 self.hidden_dim = hidden_dim
 self.num_heads = num_heads
 self.head_dim = hidden_dim // num_heads
 assert hidden_dim % num_heads == 0, "hidden_dim must be divisible by num_heads"
 
 self.q_proj = nn.Linear(hidden_dim, hidden_dim)
 self.k_proj = nn.Linear(hidden_dim, hidden_dim)
 self.v_proj = nn.Linear(hidden_dim, hidden_dim)
 self.out_proj = nn.Linear(hidden_dim, hidden_dim)

 self.k_cache = None
 self.v_cache = None

 def _split_heads(self, x):
 batch, seq_len, _ = x.size()
 x = x.view(batch, seq_len, self.num_heads, self.head_dim)
 return x.permute(0, 2, 1, 3)

 def _combine_heads(self, x):
 batch, num_heads, seq_len, head_dim = x.size()
 x = x.permute(0, 2, 1, 3).contiguous()
 return x.view(batch, seq_len, num_heads * head_dim)
 # Prefill
 def prefill(self, x):
 Q = self._split_heads(self.q_proj(x))
 K = self._split_heads(self.k_proj(x))
 V = self._split_heads(self.v_proj(x))
 self.k_cache, self.v_cache = K, V

 scores = torch.matmul(Q, K.transpose(-2, -1)) / (self.head_dim ** 0.5)
 attn = F.softmax(scores, dim=-1)
 context = torch.matmul(attn, V)
 context = self._combine_heads(context)
 return self.out_proj(context)
 # Decode 
 def decode(self, new_x):
 Q_t = self._split_heads(self.q_proj(new_x))
 K_t = self._split_heads(self.k_proj(new_x))
 V_t = self._split_heads(self.v_proj(new_x))
 
 K_cat = torch.cat([self.k_cache, K_t], dim=2) # 1..t
 V_cat = torch.cat([self.v_cache, V_t], dim=2)
 
 scores = torch.matmul(Q_t, K_cat.transpose(-2, -1)) / (self.head_dim ** 0.5)
 attn = F.softmax(scores, dim=-1) 
 context = torch.matmul(attn, V_cat)
 out = self.out_proj(self._combine_heads(context))
 
 # 最后更新 cache
 self.k_cache, self.v_cache = K_cat, V_cat
 return out


# 测试
torch.manual_seed(0)
batch_size, seq_len, hidden_dim, num_heads = 2, 100, 16, 4
mha = MultiHeadAttentionWithKVCache(hidden_dim, num_heads)
prompt_embeddings = torch.randn(batch_size, seq_len, hidden_dim)

hidden_states = mha.prefill(prompt_embeddings)
print("Prefill 完成后，cache shapes:", mha.k_cache.shape, mha.v_cache.shape)
# Prefill 完成后，cache shapes: torch.Size([2, 4, 100, 4]) torch.Size([2, 4, 100, 4])

new_emb = torch.randn(batch_size, 1, hidden_dim)
mha.decode(new_emb)
print("Decode 1步后，cache shapes:", mha.k_cache.shape, mha.v_cache.shape)
# Decode 1步后，cache shapes: torch.Size([2, 4, 101, 4]) torch.Size([2, 4, 101, 4]) 

mha.decode(torch.randn(batch_size, 1, hidden_dim))

Prefill和Decode阶段的一些优化方法 ：

DistServe： 将预填充（prefill）和解码（decode）阶段解耦，分配到不同的GPU上 ，避免两阶段在同一设备上的互相争抢资源，从而提升整体性能。

SARATHI ：针对不同长度的Prompt 导致的 Padding 冗余，采用 Chunked Prefill技术， 将较长的输入拆分为等长的chunk，每个chunk都能充分利用算力。 同时，将Decode阶段的请求则“捎带”到空闲时段并行处理。

缓解 KV cache带来的显存压力： 通过 分组查询注意力GQA 、将KV cache分布在多张GPU上存储与计算、或者建立CPU-GPU之间的KV cache调度机制，减轻单张显卡的显存压力。

KV cache复用 ：在客服、检索等高并发场景下，许多请求 共享相同的prompt 前缀，可以跨请求复用已计算好的KV cache，节省重复计算资源。

自注意力机制优化 ：比如 FlashAttention 、 SparseAttention 等高效算法。

| 对比项 | Prefill 阶段 | Decode 阶段 |
| --- | --- | --- |
| 主要功能 | 一次性并行处理完整输入序列，计算并缓存 Key/Value，用于后续生成首个及所有后续 token | 每次只生成一个新 token，利用缓存的 Key/Value，生成新输出并持续更新缓存 |
| 计算类型 | 全序列自注意力计算，高度并行、计算密集型 | 增量式注意力计算，串行生成、通信密集型 |
| GPU 利用率 | 充分利用算力资源，能高度并行，效率高 | 并行度低，通常是小批量或单 token 操作，GPU 利用率较低 |
| 计算复杂度 | O(seq_len²) | O(seq_len) |
| KV Cache 管理 | 不依赖缓存，直接基于原始输入并行计算，构建并缓存所有 prompt 的 KV cache | 每步都查 KV cache，利用其计算注意力，并将新 token 的 K_t, V_t 持续追加到缓存中 |

其中 【公式开始】Att_1(Q,K,V) 、Att_2(Q,K,V) 【公式结束】 表示注意力的第一行和第二行

![原文图片](assets/5b8b29526a3f.jpeg)

这里 【公式开始】Q_1K_2^T 【公式结束】 被mask掉了，这表明在生成第二个token“B”时，不需要对 【公式开始】Att_1 【公式结束】 重复计算。换句话说， 对于每个新的token 【公式开始】Q_k 【公式结束】 ，只需要计算 【公式开始】Att_k 【公式结束】 而不用重复计算之前的注意力值。 而在计算当前 【公式开始】Att_k 【公式结束】 时， 需要用到之前所有token的K和V，为了避免重复再计算这部分，可以将之前token计算得到的KV都缓存下来，这就是 KVcache 。

KV cache是推理阶段最常用的缓存机制， 用空间换时间 。在Prefill计算时，将计算好的K和V存储起来，在self regressive解码时，每一步解码只需要计算出当前token的QKV向量， Q向量与缓存的所有K和V进行计算，KV存储到KV cache中供后续解码使用 。 通过使用KV cache，避免了对之前token的重复计算，加速模型推理速度。

### 6.2.3 KV cache改进

MQA和GQA ：详见 1.1.3 self-attention改进

滑动窗口 ：例如 StreamingLLM 算法，

a) 密集注意力 ：时间复杂度为O( 【公式开始】T^2 【公式结束】 ),当推理文本超过预训练长度时，困惑度大幅度上升

b) 窗口注意力 ：只 维护最近的L个token的KV ，但是当序列长度超过缓存大小时， 失去第一个token的KV，会导致困惑度增加。

c) 滑动窗口与重新计算 ：为每个新的token重建最近token的KV状态（这样一直保持有初始token）。虽然它在长文本上表现良好，但它的 【公式开始】O(T L^2) 【公式结束】 复杂性(源于 上下文重新计算中的二次注意力 )使得它相当慢。

d） streaming-LLM ：保留 attention sink (注意力汇聚，汇聚在初始的几个tokens，因此保留注意力得分高的前几个token) 与最近的token结合，用于稳定的注意力计算。

![原文图片](assets/c59d1261ac5d.jpeg)

稀疏化 ：相当于更精细化的滑动窗口方法， 保留注意力得分高的token ，例如 H2O、SubGen、LESS

量化 ： 低精度保存KV cache ，例如 KVQuant， vLLM中也支持低精度KV cache。

VLLM ：利用 PagedAttention ，类似于操作系统中的页式存储或者内存分页， 允许非连续的内存空间存储连续的K和V。

KV cache卸载 ：若GPU显存不足以存储完整的KV cache，可以 将部分写在到CPU中 。

KV cache共享 ：对具有相同前缀的多条请求， 缓存并重用对应的 KV block，无需对前缀部分重复计算，几乎零成本复用。

KV cache的优点

通过缓存 Key 和 Value，解码器在生成新 token 时只需要计算当前 token 的 Query 并与缓存的 KV 交互， 计算复杂度从 O(N²) 降到 O(N) ，显著加速推理。

KV cache面临问题：

存储碎片化 ，通过pagedattention解决。

所需存储空间巨大 ，通过Multi-Query Attention解决

训练阶段不需要KV cache，因为token可以并行计算，不存在计算冗余。而推理采用自回归的方式，每次只能生成一个token，需要KV cache存储之前计算的K和V，避免重复计算并加速推理。

6. 3 vllm

推荐视频：【 怎么加快大模型推理？10分钟学懂VLLM内部原理，KV Cache，PageAttention 】

VLLM 是一个用于LLM 推理 的高性能库， 适用于需要快速响应和高吞吐量的大规模应用场景 。可以比直接调用模型的推理速度快24倍。

回顾一下之前学习过的 kv cache ：在进行自回归解码的时候，新生成的token会加入序列，一起作为下一次解码的输入。 由于单向注意力的存在，新加入的token并不会影响前面序列的计算，因此可以把已经计算过的每层的kv值保存起来，这样就节省了和本次生成无关的计算量。通过把kv值存储在速度远快于显存的L2缓存中，可以大大减少kv值的保存和读取，这样就极大加快了模型推理的速度。

分别做一个k cache和一个v cache，把之前计算的k和v存起来 ， 以v cache为例：

![原文图片](assets/8b321d9472ae.png)

这样LLM在回答时只需要计算当前token的k、v，前面token的k、v都缓存了，因此随着模型输出的不断变长，我们也没有感觉到速度变慢。

### 6.3.1 kv cache存在的问题

以13B的模型为例：（others表示forward过程中产生的activation的大小，这些activation可以认为是用完则废的，因此它们占据的显存不大）

![原文图片](assets/82fb7d4fcef0.png)

但kv cache实际的 显存利用率只有20%-40% ，大部分都被浪费了：

1、LLM在推理时，总是 按照可生成最长序列长度预分配显存 。但实际上可能并不会用到那么长。

2、LLM在推理时， 当生成第一个token时，剩下的显存已经被预分配了，这就导致无法同时响应其他的请求，不能并行处理多个请求

3、 输入的prompt的长度不同，会导致显存之间的间隔碎片 ，而这些碎片又不足以预分配给下一个文本生成

vllm就是用来解决kv cache中存在的 显存浪费问题 ， 用更大的batch size来处理请求，从而提高了系统的吞吐量 ：

### 6.3.2 continuous batching

看vllm的时候注意到了continuous batching这个技术点，按图索骥找到了最初提出它的论文《 Orca: A Distributed Serving System for Transformer-Based Generative Models 》，在这篇论文里它最初被称为 Iteration Batching。 得益于它对吞吐能力的数倍提升，现在已成为 LLM 推理框架的标配能力了。一起来回顾一下它是如何被提出，又有哪些技术细节吧！

先解释一下后面会用到的图示的意思：大模型推理过程分为两部分，如下图，其中绿色为 prompt ，蓝色为 生成的每个token ，生成过程持续直到遇到stop token或达到最长长度。

传统 static batching 存在什么问题？

Static batching指的是 按 batch 处理请求 ，每个 batch 同时生成下一个 token，直到所有请求完成。但这会带来两个主要问题：

Early-finished（提前完成）与Late-joining（迟到加入）

指的是某些请求可以在batch中很早就完成，但并未释放资源，以让新的请求加入到batch中。这意味着 GPU未被充分利用， 比如下图中，请求s1生成了4个token，s3生成了3个，s2生成了1个，s4生成了1个，seq1、2、4结束标记后的 白色方块就是GPU在空闲 ，什么都没有做，此时GPU利用率非常低，新请求也无法插队进来。同时这也意味着已经生成的短文本，要等待同一batch内其他文本生成完成才能返回给用户。

| t1 | t2 | t3 | t4 | t5 | t6 | t7 | t8 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| S1 | S1 | A1 | A1 | A1 | A1 | EOS |  |
| S2 | S2 | S2 | A2 | EOS |  |  |  |
| S3 | S3 | S3 | S3 | A3 | A3 | A3 | EOS |
| S4 | S4 | S4 | S4 | S4 | A4 | EOS |  |

Batching an arbitrary set of requests（Batch 内部异构请求难以高效处理）

另一个问题在于，如何将多个请求组合成一个 batch 进行处理，以充分利用 GPU 的并行计算能力。 然而，对同一 batch 内的请求进行处理时，要求在此次迭代中执行完全相同的操作，即每个请求待处理的张量形状需保持一致。 尽管可以通过 padding（填充）和掩码矩阵的方式解决这一问题，但这种方式会严重浪费算力，尤其是在 Batch 内部存在异构请求的情况下，高效处理会变得更为困难。

| 如右图所示：<br>x3和x4： 都在 initiation 阶段，但输入长度不同。可以通过补齐（padding）合批处理。<br>x1和x2 ：都在 increment 阶段，但生成进度不同，KV 缓存形状不一致，难以直接合批。<br>x1和x3 ：分别处于不同阶段（一个在 initiation 阶段，另一个在 increment 阶段），不能合批处理。 initiation 阶段的所有token可以并行计算， increment 阶段不能并行。 |   |
| --- | --- |
|  |  |

因为是22年的文章，这里的 initiation 指的是“请求刚进入系统、prompt 输入的预填充阶段”；而 increment 表示正式进入 decode 阶段 ，也就是每次生成下一个 token 的过程。

综上所述， 只有处在相同阶段、且输入或张量形状完全相同的请求才能同一个batch处理 。

Continuous Batching 如何解决以上两个问题？

对于第一个问题的解决方法就是 一旦一个batch中的某个请求完成生成，发射了一个end-of-seq token，就可以在其位置插入新的请求继续生成token，从而达到比static batching更高的GPU利用率。 而之所以能这样实现， 是因为一次推理只生成一个token，利用两次推理之间的空隙，就可以进行请求增减和显存分配。 如下图所示，请求S1、S2、S4被处理完毕后，马上开始处理新的请求。

对于第二个问题continuous batching采用 Selective batching 方法。

哪些运算对输入形状没要求（比如 Linear 层），就把所有请求拼一起高效计算；哪些运算必须对每个请求单独处理（比如 Attention），就分别算，保证计算正确。

非 Attention 运算： 诸如 线性变换、LayerNorm、加法和 GeLU 等运算， 不需要进行Token之间的交互，也就不需要区分出不同请求 ，因此可以将多个请求的张量先进行拼接，比如下图中x3和x4可以拼接为成一个形状为 【公式开始】[ \sum L, H ] = [5, H] 【公式结束】 的二维张量，再送入GPU进行计算 。图中QKV Linear层的输入和输出分别是 【公式开始】[7, H]、[7, 3H] 【公式结束】 ，因为用一个Linear层计算出了QKV3个矩阵，故为3H。

Attention 运算： Attention 要求只能在同一个请求内计算token之间的关系， 因此在进入 Attention 前插入一个 Split 操作，分别计算每个请求的Attention；随后再通过 Merge 操作将各子结果重新合并 ，以便后续运算继续使用批处理 。

### 6.3.3 Page d Attention

vLLM 采用一种 集中式调度器（scheduler） 来协调分布式 GPU 工作器（worker）的执行。 KV 缓存管理器 由 PagedAttention 驱动，能以 分页 方式有效管理 KV 缓存。具体来说， KV 缓存管理器通过集中式调度器发送的指令来管理 GPU 工作器上的物理 KV 缓存内存。

![原文图片](assets/fc93250c7211.png)

回忆 操作系统中的虚拟内存和页管理技术 ，需要解决以下问题：

操作系统给每个程序怎么分配内存？

要不要预分配内存？

程序关闭后怎么回收内存？

内存碎片怎么处理？

怎么管理内存？

虚拟内存 以页为最小单位来分配内存，物理内存被划分为很多页。

![原文图片](assets/c10454c0bfd9.png)

PagedAttention 允许 在非连续的内存空间 中 存储连续的键和值 。具体来说， PagedAttention 将每个序列的 KV 缓存划分为块，每个块包含固定数量 token 的 key 和 value ：

![原文图片](assets/0e67a21f9309.png)

例如，每个kv block可以缓存4个token的k、v向量

1、 按需分配 ，不提前预分配

2、 按block分配 ，解决了显存碎片的问题（因为碎片最大也只有3个token的大小）

3、 虚拟内存 ：维护了逻辑kv cache，让系统看上去是使用的连续的显存 （ 逻辑连续，物理不连续 ） ， 通过映射表来实现，方便block的调用。

![原文图片](assets/15b4f830b0d1.png)

对 KV 缓存的请求会被表示成一系列逻辑 KV 块，在生成新 token 和它们的 KV 缓存时从左向右填充。最后一个 KV 块中未填充的位置留给未来填充。 通过page d attention的优化， kv cache的显存利用率从20%-40%提升到了96% ，因为浪费只会发生在最后一个块中。采用vLLM能将更多序列进行批处理，提高 GPU 使用率，显著提升吞吐量。

PagedAttention的优点

页按需分配： PagedAttention 将每个请求的 KV Cache 拆分为若干固定大小的页，并 仅在生成新 tokens 时按需分配物理页，无需提前为最大序列长度预留内存，从而大幅降低 KV Cache 内存占用 。

跨请求 KV Cache 共享： 相同前缀的多个请求可共用 KV pages ，大幅降低内存占用，适用于beam search、A/B test等场景。

消除zero-padding： 传统方法往往需将所有序列填充到同一最大长度，导致短序列浪费大量内存和计算。 PagedAttention 只在实际需要的 blocks 上分配内存，短序列不会因对齐而被额外填充，从而 zero-padding 问题被根本杜绝。

提升推理吞吐： 由于减少了内存浪费和碎片，GPU 可以在同等显存下加载更多并发请求 ，vLLM 的实测表明 PagedAttention 能带来 2–4× 的整体吞吐提升，且在中长序列下优势更明显。

## 6.4 推理评估

参考文章：《 LLM inference speed of light》《 https://arthurchiao.art/blog/llm-inference-speed-zh/ 》

### 6.4.1 推理机制

推理方式 ：语言模型生成文本时（也就是Decode阶段），是逐个token生成的，无法并行。

生成过程建模 ：一般采用kv cache

瓶颈分析 ：需要每个key执行一次 乘加运算 ，对每个 value 执行一次 乘加运算。而ALU的速度要比访问内存快得多。

### 6.4.2 推理时延

总数据量

以Mistral-7B举例，该模型约有7111M参数，采用FP16计算精度，每生成一个token，需要加载到 算术逻辑单元（ALU） 的数据量为：14.2GB。

虽然计算下一个token时的参数可以复用，但硬件缓存的大小通常只有几十 MB， 参数无法放入缓存中，因此 生成（推理）过程的速度不会快于显存带宽。

attention 计算需要读取当前 token 及前面上下文中所有 tokens 对应的 KV-cache， 所以 读取的数据量 取决于 生成新 token 时模型看到多少前面的 token ， 这包括

系统提示词 （通常对用户隐藏）

用户提示词

前面的模型输出

可能还包括 长聊天会话 中多个用户的提示词。

KV cache数据量

对于Mistral-7B，KV-cache，为每层的每个 key 存储 8 个 128 元素向量，为每个层的每个 value 存储 8 个 128 元素向量。这加起来，每个 token 对应 32 * 128 * 8 * 2 = 65K 个元素； 如果 KV-cache 使用 FP16，那么对于 token number P，我们需要读取 P * 130 KB 的数据。 例如， token number 1000 将需要从 KV-cache 读取 130MB 的数据。 跟 14.2GB 这个总数据量相比，这 130MB 可以忽略不计了。

推理时延

以NVIDIA RTX 4090 （1008 GB/s）举例， 14.2GB的参数需要约14.1ms读取 。这是理论下限，代表了生成每个 token 的最小可能时间。

分析理论下限有什么用

如果实际推理时延与理论推理时延存在较大差异，往往是需要检查代码和硬件了。

推理过程并未充分利用ALU算力，可以通过 数据并行，异步通信与计算重叠 等方式。

带宽与每个权重使用的 bit 数成正比 ，对于低延迟场景，可以考虑更小的权重量化。

### 6.4.3 压测

大模型压测是什么？

大模型压测指是针对LLM及其服务系统进行的压力测试。 它的核心目的是 评估和验证大模型服务在接近或超过其设计负载极限情况下的性能、稳定性和资源利用情况 。压力测试通过在超出正常使用条件下模拟高并发、大负载场景，能够全面评估系统在极端情况下的表现和稳定性，为保障服务质量、发现性能瓶颈、进行容量规划、优化成本以及验证系统架构和优化效果提供科学依据。

可以把它简单理解为： 模拟大量用户同时向大模型服务发起请求，看看这个系统会不会“崩”、反应有多快、能撑住多少并发量 。

为什么要做压力测试？

确保服务质量 ： 在用户量激增或突发流量冲击时，如果缺乏压力测试，就无法保证系统依然能快速、稳定响应，从而导致用户体验急剧下降或服务中断。压力测试能够模拟高并发请求，对系统的响应时间、吞吐量和错误率进行监控，确保即使在最糟场景下也能维持可接受的服务质量水平。此外，通过在测试阶段提前暴露潜在崩溃风险，可以避免线上环境出现的严重故障。

发现性能瓶颈 ：压力测试将系统推向极限，能够精准定位系统性能的“最弱一环”，例如模型推理延迟、数据库查询慢、CPU/GPU 资源耗尽或内存泄漏等问题。通过对各项性能指标（如响应时间、吞吐量、资源利用率）的实时监控和日志分析，开发团队能够快速识别并优化关键路径，显著提升系统的整体性能和稳定性。

容量规划 ：了解系统在不同负载下的最大承载能力，是进行合理资源配置的前提。压力测试可以确定系统能够同时处理的最大用户数或请求数，帮助团队制定容量上限、预留冗余资源，并据此编制 SLA（服务等级协议）和预案。通过这些数据，运维团队能够避免过度采购资源导致浪费，也能在流量高峰前动态扩容，保证系统平滑运行。

成本优化 ： 在云环境或资源计费模型下，资源消耗直接对应成本。压力测试提供了不同负载下的 CPU、GPU、网络和存储等资源使用情况，让你能够基于真实数据选择更合理的实例类型和自动伸缩策略，避免过度预留与资源浪费。同时，结合持续监控和自动化工具，可以实现实时成本管控，在不影响性能前提下最大化资源利用率，从而降低整体运营成本。

验证系统架构和优化效果 ：在进行架构调整（如引入缓存、服务拆分、异步处理）或性能优化（如模型量化、数据库索引优化）后，必须通过压力测试来验证改动是否达到预期效果。只有在同一负载场景下对比优化前后的指标变化，才能科学评估优化成果，避免盲目迭代带来的回归或新问题。通过持续集成流水线中纳入压力测试，还能保证后续版本迭代持续满足性能和稳定性要求。

以下是压测时应该重点关注的指标：

性能指标 ：

延迟 ：

平均延迟 ： 客户端发送请求到收到完整响应所消耗的平均时间 。

尾部延迟（P90/P95/P99） ： 分别表示 90%、95%、99% 的请求响应时间低于该值 ，用于评估极端情况下的响应性能。

吞吐量 ：QPS/TPS， 系统在单位时间内处理的请求数或事务数 。

并发用户数 ：模型服务能够 同时承载的活动会话或并发请求数 ，直接关联架构的并行处理能力。

系统稳定性与可靠性

错误率 ： 在高负载下产生的超时、异常或错误码的比例 ，反映服务的鲁棒性与容错设计。

崩溃/宕机检测 ： 验证在极限压力条件下服务是否发生进程崩溃或自动重启 ，评估故障恢复能力。

长时间稳定性 ： 持续在高压环境下运行数小时至数天 ，观察性能指标是否漂移或出现内存泄漏等问题。

资源利用率

CPU/GPU 使用率 ： 监控计算资源的占用率 ，识别是否存在计算瓶颈或资源闲置。

内存/显存使用率 ：防止因超出内存上限导致的 OOM （Out Of Memory）错误，同时为动态批处理或缓存机制提供决策依据。

网络带宽与磁盘 I/O ： 评估数据传输与模型加载过程对网络和存储的压力 ，尤其在分布式部署或远程模型拉取场景下。

可扩展性

水平扩展性能 ： 在增加服务实例或 GPU 卡数后，吞吐量与延迟指标的提升比例，用于验证部署架构的横向扩展效率 。

动态伸缩能力 ：结合自动伸缩策略测试系统 对突增流量的响应速度与资源调整准确性 。

## 6.5 推理重要论文

### 6.5.1 Distserve（PD分离）

LLM 推理分为两阶段：

Prefill 阶段 ：在这个阶段，模型并行处理用户输入的Prompt，并生成第一个输出 token。这个过程的延迟一般用 TTFT 来衡量。 Prefill阶段是计算密集型任务 ，对计算资源的消耗很大。即使batch-size不大，但如果用户的输入很长，依然可能使得GPU的计算能力达到极限。

Decoding 阶段 ：进入这个阶段后，模型会基于已生成的 token ，逐步生成接下来的每一个 token。这个阶段的延迟通常用 TPOT 来衡量。 Decoding计算是通信密集型任务 ，每步只处理一个新token，只有更大的batch-size才能充分利用GPU算力，但受到GPU内存带宽的制约（因为KV-Cache的读取与存储）。

首先介绍一下Distserve这篇论文中评价LLM服务系统性能的几个指标：

时延（Latency）： 时延是指用户从发起请求到收到响应所等待的时间，包括：

首词元输出时间（TTFT ， time-to-first-token ） ： 生成第一个 token 的等待时间 。

每词元输出时间（TPOT ，time-per-output-token ） ： 连续生成 token 之间的平均时间间隔 。

吞吐量 （ Throughput/rps ）：吞吐量常用 rps（Requests Per Second） 来衡量，指 系统每秒能处理多少个用户请求 。 吞吐量越高，单位成本越低 。但如果只追求吞吐量，可能会牺牲用户的实际体验。为此引入了SLO的概念：

服务等级目标 （ SLO ，Service level objective）： SLO 是针对 TTFT、TPOT 等时延指标设定的具体目标要求，比如“90% 的请求 TTFT < 200ms，TPOT < 50ms”。它用于保证大多数用户的体验不被影响。

有效吞吐量 （ Goodput ）：指 在 SLO 要求下 ，系统每秒实际能处理的请求数量。只有同时保证高吞吐和良好体验的请求才计入 Goodput。

造成这一现象的原因：

Continuous Batching带来的延迟累积

为了提升整体吞吐量，现有系统通常采用continuous batching策略，对大量请求同时进行 prefill 与 decode。 这种方式确实能最大化 GPU 利用率，但会导致部分请求在 decode 阶段等待过久，违反 SLO ，从而不计入 goodput。如下左图中decode任务(R1)因为中途插入了prefill任务（R2），导致decode时间大幅拉长。

Continuous batching为了实现并行计算，batch 内所有请求必须同步进行同一操作，哪怕其中部分请求只需 decode，也得等待 batch 内其它请求的 prefill 结束。

![原文图片](assets/c06a0620c09d.png)

简单拆分prefill和decode也难以解决 ： decode任务可能需要等待进行中的prefill任务，专门做decode的batch的的GPU利用率也很低 。

计算资源和并行方式耦合，难以兼顾所有延迟需求 ：因为prefill和decode部署在同一块GPU上，所以要共享计算资源和并行策略。但 prefill阶段受到算力限制，为了满足TTFT要求，更需要做张量并行（TP）以缩短执行时间；decode阶段更适合大 batch 的流水线并行（PP），以提升整体 TPOT 和吞吐量。 现行的推理系统中，为了同时满足两个SLO，往往需要配置过度的计算资源。

预填充和解码分析

prefill阶段是计算密集型任务， 一旦待处理请求的输入长度超过某个阈值 【公式开始】L_m 【公式结束】 ，GPU 即成为计算瓶颈，此时继续增大batch-size不仅无法提升吞吐率，反而会按比例延长每个batch的处理时间，应保持较小batch-size以避免过度排队。

![原文图片](assets/a4e76bbbeaef.jpeg)

论文中对不同并行策略平均TTFT的计算 ：

假设预填充采用 M/D/1 排队系统，当请求到达率为 R、单请求执行时间为 D 时， 【公式开始】RD < 1 【公式结束】 ，TTFT的平均值为：

【公式开始】Avg\_TTFT = D + \frac{RD^2}{2(1 - RD)} 【公式结束】

引入2路张量并行后，设加速系数为 【公式开始】K(1 < K < 2) 【公式结束】 ，因为通信开销会浪费一定的时间，TTFT的平均值为：

【公式开始】Avg\_TTFT = \frac{D}{K} + \frac{RD^2}{2K(K - RD)} 【公式结束】

引入2路流水线并行后，忽略跨层通信的开销，设整体请求延迟为 Dₛ，最慢阶段耗时为 Dₘ， 【公式开始】D ≈ Dₛ ≈ 2 × Dₘ 【公式结束】 ，TTFT的平均值为：

【公式开始】Avg\_TTFT ≈ D + \frac{RD^2}{4(2 - RD)} 【公式结束】

### 6.5.2 Sarathi 之 Chunked-prefill

背景介绍

在基于 Transformer 的大模型推理中，整个生成过程可分为两大阶段：

预填充（Prefill）阶段 ： 对完整的输入提示（prompt）一次性并行计算 ，prefill阶段计算量较大，batch-size较小就能充分利用GPU计算资源。

解码（Decode）阶段 ： 自回归的逐步生成新 Token ，每次只处理一个token，如果batch-size比较小，GPU利用率很低。并且对于每个请求，要进行一次prefill和多次decode，这会进一步降低推理效率。

为了提升Decode阶段的效率，主流做法是通过张量并行或流水线并行+micro-batch提升batch-size，但前者通信开销高，只有在高带宽通信网络下（如NVlink）才能使用，适用场景受限。后者会导致流水线气泡，因此提出 SRATHI 算法 提升解码效率并缓解流水线气泡 ：

Chunked prefill ： 将一个用户问题的prefill阶段拆成多个计算量相同的chunk，每个chunk都能单独饱和 GPU 运算单元 。另外每个块都独立计算，从而保障能构建足够的混合batch， 提升了与Decode请求捆绑执行的机会。

Decode-Maximal Batching ： 将一个Chunked-Prefill计算块与尽可能多的Decode请求打包到同一batch中 。

由于Chunked-Prefill块本身已经极大地饱和了GPU， 额外加入多个Decode请求仅需极少额外开销（相较于纯Decode的batch），并且可以通过复用相同Chunked-Prefill块的KV cache，从而提升Decode效率 。

统一的计算量则显著减少了流水线气泡。下图展示了2路-流水线的情形。图（a）中，第一个气泡是因为不同的用户查询长度，GPU1在跑完 【公式开始】D_p 【公式结束】 后，需要 等待GPU2prefill完 【公式开始】A_p 、B_p 【公式结束】 的后半截模型 ，并把所需的KV cache传回给GPU1才能开始第一次解码 【公式开始】A_{d1}B_{d1} 【公式结束】 。第二个气泡是因为prefill和decode计算时间不匹配，在缺少micro-batch调度的情况下，GPU1解码完 【公式开始】A_{d1}B_{d1} 【公式结束】 要 等待GPU2完成 【公式开始】C_pD_p 【公式结束】 的prefill阶段的后半截模型，才能开始解码 【公式开始】C_{d1}D_{d1} 【公式结束】 。

图（b）中，首先将完整的prefill拆分为多个chunk，然后实现在多GPU之间的交错调度。具体来说，GPU1上 【公式开始】C_{p1} 【公式结束】 执行完成后，chunk内还有额外的计算时间，可以搭载已经prefill完的请求A，执行第一次decode 【公式开始】A_{d1} 【公式结束】 ， 由于Chunked-Prefill块已经极大地饱和了计算资源，额外加入Decode 几乎不额外占用时间 。换句话说，Chunked-Prefills 产生了更多、更均匀的“填充段”，可以将Decode插入这些段的间隙中，填满原本 Prefill 与 Decode 切换时的空隙

LLM推理低效源头

LLM推理低效的两个原因：

Decode阶段的通信瓶颈 （原文中的表述是 memory-bound ，我理解指的是内存带宽限制）。

流水线并行调度导致的 流水线气泡 。

Decode阶段memory-bound

| Prefill 阶段在batch-size(B), sequence length(L) 满足 【公式开始】B×L≥512 【公式结束】 时，就能达到巅峰吞吐量180 tokens/ms， 增大batch-size对吞吐量没有影响 。Prefill阶段的输入是 【公式开始】[batch\_size, sequence\_length, hidden\_size] 【公式结束】 ,执行的是矩阵-矩阵乘法。 | Decode 阶段，只有当batch-size极大时（L=1024时，B>256）才开始接近饱和，但 这么大的batch-size会导致KV cache占极大显存，无法实现 。Decode阶段的输入是 【公式开始】[batch\_size, 1, hidden\_size] 【公式结束】 ,执行的是向量-矩阵乘法。 |

定义 算术强度 为： 每次内存读写可使用的计算量 。如下图所示， Prefill阶段即使batch-size为1时算术强度也很高，能充分利用GPU算力（ compute-bound ）。而在Decode阶段，只有batch-size很大时，算术强度才能比较高，但大batch-size会增大KV cache的存储和读取压力，算术强度比较低（ memory-bound ）。

![原文图片](assets/637f43152000.jpeg)

流水线气泡

请求级调度 （request-level scheduling）：如 FasterTransformer 中， 将一批请求打包后一次性执行 Prefill 和 Decode ， 导致短序列请求被pad到最长长度 ，浪费计算资源。

迭代级调度 （iteration-level scheduling）：如 Orca、vLLM、HuggingFace TGI 等， 一组请求每次生成一个新token，允许请求在每次迭代开始／结束时动态进出批次，仅保证批大小恒定。

OCRA(即最早提出continuous batching的论文)中的iteration-level scheduling，仍会导致推理阶段出现流水线气泡，主要有三种原因（前两种之前讨论过了）：

PB1：Prefill 长度差异， 相邻micro-batch中需要预填充的token数不同， 导致prompt短的需要等待prompt长的 。

PB2：Prefill和Decode 时长不匹配。 同一micro-batch中不同GPU分别进行prefill和decode， 计算快的decode要等待计算慢的prefill 。

PB3：Decode KV cache长度差异。 不同请求的 KV cache 深度不同 ，使得不同请求的Decode时间不一。

![原文图片](assets/87ca48057872.jpeg)

SARATHI：Prefill多趟车+Decode搭便车

SARATHI的核心思想为：将大 Prefill 切分为等算力的小块（ chunked-prefills ），每块都能饱和GPU算力的同时，与尽可能多的 Decode 请求混合成同一微批次（ decode-maximal batching ）。这样既能 保持高GPU利用率 ，又能 将Decode的高KV cache压力摊薄到多块中 ，同时 消除各micro-batch之间的时长差异，从根本上减少流水线气泡 ，提高整体吞吐。

Chunked-Prefill

核心思想：将一次完整的 Prefill切分成多个等算力的chunk，使得每个chunk都能单独饱和GPU计算。

之所以可以将prefill进行分块，主要基于两个关键洞见：

Prefill 吞吐饱和点 ：不同模型/显卡组合下， Prefill 对 token 数的吞吐量随着输入序列长度增长，在某个阈值 （如上上上图中 LLaMA-13B+A6000 在 512 token 处） 便趋于饱和； 选用比饱和点略小的 chunk（如 256）只损失约 12.5% 的吞吐，却能带来更细的调度粒度。

实际 Prompt 足够长 ： 生产环境中常见的prompt在 1K–4K 之间 （这篇论文写于2023年，近期随着推理模型的火热，大多数模型支持的上下文长度也在变得越来越长）， 有足够空间拆分出多个 chunk 。

Chunked-Prefill的注意力mask 矩阵设置如下图所示， 当前chunk中的每个token可以访问之前的所有token，不能访问之后的token ，保证分块计算与一次性prefill在数值上完全等价。

![原文图片](assets/7a3c2e1c7a69.jpeg)

引入Chunked-Prefills 的开销：每个 chunk 都要重新读取前面所有已生成 token 的 KV 缓存 。

但是，作者通过实验验证了以下观点：

注意力计算在一次前向传播中耗时占比小 。KV cache只在注意力计算时使用，而 注意力计算在前向传播中占比很小，主要时间都消耗在FFN层 。并且考虑到prefill阶段是 compute-bound 任务，主要时间耗费在计算上，增加通信时长影响不大。

单看Chunked-Prefill这一个阶段，当chunk比较小时（64），对注意力计算和整个prefill阶段都会造成很大的速度损失，但 当chunk提升到256/512时，损失就只有20%和10%了 。而如果 结合上decode-maximal batching来看，Chunked-Prefill在chunk大于128时都能提升吞吐量，因为可以携带更多得Decode任务（Piggybacking Decodes）。

![原文图片](assets/d0720fb85258.jpeg)

Decode-Maximal Batching

核心思想：每个batch选取一个Prefilled-chunk，再用尽可能多的 Decode 任务填满剩余的slots。 这样既保证了batch的算术强度（Prefilled-chunk），保障每个batch的计算量相同，又能将多个 Decode 请求“搭载”进来，摊薄它们的内存带宽开销。

首先要 确定能搭便车的最大Decode任务数 。给定可用 GPU 显存 【公式开始】M_G 【公式结束】 、模型参数占显存大小 【公式开始】M_S 【公式结束】 ，最大序列长度 【公式开始】L 【公式结束】 ，每个token占用的KV cache 【公式开始】m_{kv} 【公式结束】 ，最大Decode任务数为： 【公式开始】B - 1=\lfloor (M_G - M_S)/(L\times m_{kv})\rfloor - 1 【公式结束】 ，(减去一个prefilled-chunk）。

对于注意力运算，Decode和Prefilled-Chunk分别计算（因为输入形状不统一）， 线性计算对输入形状没要求，为了提升解码效率就把所有Decode和Prefilled-Chunk拼一起高效计算 。 线性计算中，Decode和Prefill阶段使用相同的权重张量，将Prefill-chunk和Decode的线性计算合并成一次矩阵-矩阵乘运算，只需要加载一次模型权重 。如表2所示，纯 Decode 阶段每token平均耗时约12.49 ms；而在Decode-Maximal Batching 中，每token耗时仅1.2 ms，约 10× 加速。

如何确定Prefilled-chunk大小

定义 P : D比 为 某个混合批次中 Prefill token 数 P 与 Decode token 数 D 的比值 。

举例：batch size = 4（1 个 Prefill + 3 个 Decode），若 chunk size = 128，则一个 Prefill 长度为 P 的请求可分出 P/128 个 chunk，每个 chunk 最多搭载 3 个 Decode，共能覆盖约 P/42 个 Decode；要完全覆盖，则需 P:D＞42。

Chunk size小会导致P:D小， 虽然能携带更多的Decode，但会降低prefill效率 （因为chunk size小可能无法充分利用GPU算力，并且增加了Prefilled-chunk访问KV cache的次数）

反之，Chunk size大，会减少能携带的Decode。

理想的chunk size取决于目标场景的平均P:D，以及Prefill 与 Decode 在该场景下的时间分布。 (如果Decode占用了90%的时间，Prefill占了10%的时间，那为了Decode能加速2倍，prefill效率降低5倍也是值得的)。

此外，由于GPU执行矩阵乘法时，会将矩阵划分为若干tile并行计算。如果矩阵维度不是tile大小的整数倍，那就会浪费算力。因此， prefilled-chunk和携带的Decode token之和应该是tile大小的整数倍 。

大模型推理可以分为两阶段：Prefill和Decode，前者预填充可以并行执行效率很高。本文主要关注后者。

自回归解码 ： 每次解码只生成一个token，最传统的自回归解码想要生成N个token，需要执行N-1次前向传播， 推理延迟随生成序列长度和模型规模线性增长。虽然通过KV cache能避免重复计算之前的token，但会带来很大的显存压力。

Continuous batching ： 即已经解码完毕的任务可以退出batch，允许新的请求插入到batch中 ，减少了因等待导致的算力浪费。但在用户请求少时无法发挥作用。

Parallel Decoding ：通过 Blockwise Decoding ， 在Transformer解码器顶部添加额外的FFN头，使得每步推理可以同时生成多个token 。 这些token随后由原始LLM并行验证 ，接受第一个不一致的token之前的所有token，确保输出与原始LLM的输出一致。

投机解码 ： 每次解码先用轻量的 非自回归的drafter模型 预测多个未来的token，然后使用LLM并行地验证，从而加速推理 。只有符合LLM验证标准的token才会被接受为最终输出，以保证生成质量。

许多 简单的token使用简单的模型就能预测 。

解码是 通信密集型任务 ，每次解码都需要将大量参数从HBM加载到SRAM中， 利用投机解码可以减少解码次数，从而减少通信次数 ，从而提升推理效率。

### 6.6.2 Drafting

预测的关键在于在 预测精度 和 预测时延 之间找到平衡。

Independent Drafting

思想 ： 用一个与目标 LLM 不同、更高效模型 来并行生成多步候选分布，主要方法分为：

专用非自回归 Transformer： 如 SpecDec 中的deep-shallow encoder-decoder结构（即多层编码器，少量解码器），需要额外训练 。

同系列小模型： 无需额外训练 ，因共享 tokenizer和预训练数据，且模型结构类似，同系列小模型天然就与目标大模型更加对齐。为了进一步提升对齐效果，还可以通过知识蒸馏等手段训练小模型。

Self-Drafting

思想 ： 在 目标 LLM 内部 进行预测 ，避免额外训练或调度小模型带来的额外开销。

FFN Heads： 如Block-wise Decoding和Medusa， 在 Transformer 解码器上添加多个额外的并行 FFN 头，模型可以在每次解码都并行生成多个token ，开销小，易于分布式部署。

Early Exiting & Layer Skipping： 如 Self-Speculative， 通过提前退出当前层或跳过中间层来快速生成后续分布。

Mask-Predict & N-grams： 直接在输入提示的末尾追加多个 [PAD] 标记，利用并行掩码填充实现并行生成，将低质草稿转为 n-gram 片段提升质量。

Learnable Tokens： 引入可训练的特殊 [LA] token，通过小规模微调改善并行预测效果。

### 6.6.3 Verification

贪婪解码

早期的投机解码大都采用贪婪解码，以保证输出与目标模型的贪婪解码完全相同。给定输入序列 【公式开始】x_1, \dots, x_t 【公式结束】 和drafted模型预测的 token 【公式开始】\tilde{x}_1, \dots, \tilde{x}_K 【公式结束】 ，以及目标模型 【公式开始】M_q 【公式结束】 预测的概率分布 【公式开始】q_1,...,q_K 【公式结束】 ，

严格匹配（Lossless） ： 要求每个drafter模型预测的token恰好等于目标模型 【公式开始】M_q 【公式结束】 的 Top-1 预测 ，即 【公式开始】x̃ᵢ = arg max qᵢ 【公式结束】 。 若第c个预测token不符，则以目标模型的 Top-1 结果替换，并舍弃后续预测的token ，代表工作如 Blockwise Decoding。但是 严格匹配可能导致高质量、但跟目标模型的Top-1不同的token被拒绝，限制投机解码的加速效果。

近似松弛（Approximate） ：为了提高接受率，可以 放宽匹配要求 。例如SpecDec允许drafter模型预测的token在目标模型的 Top-k 结果中出现，Bild 只在连续不匹配的token数量超过固定阈值时拒绝token。

### 6.6.4 Alignment

如前文所说， token接收率直接决定了投机解码的加速效果，而接受率高低关键在于 drafter 模型的输出分布与目标模型分布的相似度 ，可以通过知识蒸馏的方式提升两个模型的对齐效果。

序列级知识蒸馏（Seq-KD）： 在目标模型生成的序列上训练drafter模型。

集体增强微调（Col-BT）： 应用Seq-KD对多个小模型进行微调，利用它们的聚合输出作为预测，以提高预测的准确性。

在线知识蒸馏（Online KD）： 在推理过程中基于查询数据动态更新 drafter，无需预先训练，即时对齐目标模型 。

### 6.6.5 投机解码应用：Medusa

本节介绍一下Medusa这个方法，通过这篇论文学习树注意力机制。

Medusa遵循和投机解码相同的框架，可以分为三个步骤：(1) 生成候选，通过Medusa 头实现；(2) 处理候选，通过树注意力实现； (3) 接受候选，通过拒绝采样或者典型接受实现。

MEDUSA 头

其中 【公式开始】d 【公式结束】 表示隐藏层的维度，V是词表的维度。 【公式开始】W^{(k)}_2 【公式结束】 初始化为原语言模型head权重， 【公式开始】W^{(k)}_1 【公式结束】 初始化为0，保证MEDUSA 头的初始预测与原始模型一致 。MEDUSA的头部跟主干模型一起训练，能利用到大模型的强大表征能力，并且确保MEDUSA头部的分布与原始模型一致，减轻分布偏移问题。

Tree Attention

通过MEDUSA头获取了后续K+1个预测token， 传统的投机解码方法每个头只采样一个候选 。 MEDUSA为每个头生成多个候选token，以提升预测token的接受率（从top1变成了top5，自然预测正确的概率就高了） ，为了平衡计算量，采用树注意力来同时处理多个候选。

对于第k个头，预测 【公式开始】top 【公式结束】 - 【公式开始】s_k 【公式结束】 个token，然后根 据每个头生成预测的笛卡尔积来构建出多个序列，形成树状候选空间 。如左图所示， 【公式开始】s_1= 2 【公式结束】 和 【公式开始】s_2 = 3 【公式结束】 ，构成了6个候选序列，其对应注意力掩码矩阵如右图。Token 树中的token数为 【公式开始】\sum_{k=1}^K \prod_{i=1}^ks_i = 8 【公式结束】 ，虽然解码阶段的计算量比较低，但也不意味着可以无限增加要生成的token数。

稀疏树

每个Medusa 头都取top-k，然后做笛卡尔积构建树是一种最简单的实现（稠密树），但某些Token准确率很低，计算这些Token会浪费算力，特别是在节点数受限的场景。

因此作者提出 在校准集上统计每个Medusa 头的top-i候选的准确率 ，第k个头的第i名准确率记为 【公式开始】a^{(i)}_k 【公式结束】 。假设各个Medusa 之间独立，一个由 [i1, i2, ..., ik] 组合成的候选token序列，其整体准确率约为 【公式开始】\prod_{j=1}^{k} a^{(i_j)}_j 【公式结束】 。

目标： 给定节点数预算，构造一棵能 最大化候选被接受长度期望 的最优稀疏树。

枚举计算所有候选序列的准确率，再 贪心地把最高准确率高的分支优先加入树，直到节点数用完 。

这种方法能够得到一棵“稀疏但高效”的树，显著提升树中被采纳的平均prefix长度，从而提升推理效率和质量。

训练策略

假设有与原始模型输出分布一致的数据集（一般为SFT用的同源数据，没有分布一致的数据集可以通过以下介绍的自蒸馏方法获取），为了提升MEDUSA头预测未来token的能力，有两种方案：MEDUSA-1（ 只训练解码头 ）和MEDUSA-2（ 主干与解码头一起训练 ）。

MEDUSA-1： 利用MEDUSA头和真实值之间的交叉熵损失，设在第 【公式开始】t+k+1 【公式结束】 处的真实token为 【公式开始】y_{t+k+1} 【公式结束】 ， 【公式开始】p_t^{(k)}(y) 【公式结束】 为第k个头预测为y的概率，第k个头预测的损失为 【公式开始】L_{k} = \sum_{k=1}^{K} -\log p^{(k)}_t(y_{t+k+1}) 【公式结束】 。 k越大预测越不稳定，因此预测的损失越大，因此在损失中添加权重 【公式开始】\lambda_k 【公式结束】 以平衡各个头 （实际中用 0.8 的k次幂）：

【公式开始】L_{MEDUSA-1} = \sum_{k=1}^{K} -\lambda_k \log p^{(k)}_t(y_{t+k+1}) 【公式结束】

为了进一步加速训练效率和节省显存空间，还可以对主干模型进行QLoRA量化 。

MEDUSA-2： 为了进一步提升MEDUSA头的预测精度，可以 将主干与MEDUSA头一同训练 ，为了保留主干模型预测下一个token的能力和预测精度，采用了以下策略：

组合损失 ： 将主干模型的交叉熵损失加到MEDUSA损失中 ：

【公式开始】L_{MEDUSA-2} = L_{LM} + \lambda_0 L_{MEDUSA-1} 【公式结束】

不同的学习率 ： 主干模型学习率低而MEDUSA头学习率高 ，促使MEDUSA 头更快收敛且主干保持原有能力。

Medusa头预热 ：由于MEDUSA头中 【公式开始】W_1 【公式结束】 初始化为0，训练初期 【公式开始】L_{MEDUSA-1} 【公式结束】 比较大，导致较大的梯度并可能破坏主干模型。因此 采用两阶段训练，第一阶段只训练MEDUSA头，第二阶段使用warmup策略一起训练 。

通过以上策略， MEDUSA-2可与SFT流程无缝融合，获得“原生支持MEDUSA”的LLM 。

MEDUSA头数：最多5个头。

典型接受

在MEDUSA框架下，每步会并行生成多个候选序列，接下来的问题就是： 如何在众多候选序列中判断哪些可以被采纳？

传统的投机解码论文采用 拒绝采样 ，即 先生成一批候选token，然后用原始模型“验收”每个候选序列，只有通过验收的才被采纳，从而保障输出不会偏离原始模型本来的分布 。拒绝采样的问题在于，一方面引入了额外的开销，尤其是drafter模型和原始模型分布差异大时，另一方面为了保障多样化的输出，不应该要求完全匹配原始模型的分布。

典型接受（Typical acceptance） 不要求输出分布与原模型一模一样，而是在保证多样性和高效性的前提下， 优先接受“原始模型也认为合理”（生成概率不是极低）的典型序列 。具体而言，给定上下文 【公式开始】x_1, \dots, x_t 【公式结束】 ，对于候选token序列 【公式开始】x_{n+1},...,x_{n+K+1} 【公式结束】 ，依次计算每个token在原始模型下的概率，满足以下条件即可被接受：

【公式开始】p_{\text{original}}(x_{n+k}|x_1, ..., x_{n+k-1}) > \min\big(\epsilon, \delta \exp(-H(p_{\text{original}}(\cdot|x_1, ..., x_{n+k-1})))\big) 【公式结束】

【公式开始】H(\cdot) 【公式结束】 是原始模型的熵函数， 【公式开始】\epsilon、\delta 【公式结束】 是阈值参数，这就保证了 概率高的token和熵比较高的序列都会被接受 。并且为了保证每一步至少生成一个 token，我们 对第一个 token 使用贪婪解码并无条件接受它，而在后续 tokens 的处理中采用典型接受 。如果某个token不满足条件，则后续序列都被拒绝，继续验证下一个候选， 每一步最终的预测由所有候选中接受的最长前缀决定 。

典型接受总结

相较严格的拒绝采样， Typical Acceptance更宽容，速度更快，同时输出质量相似 。只要序列在原始模型下“不是很奇怪”，即可被快速采纳，不会引入无意义/低质量token。并且无需严格复现原始模型分布，能适应分布波动，容忍一定偏差，利于工程部署和推理加速。

自蒸馏

在没有与原始模型输出分布匹配的训练数据集的情况下（例如未开源原始的SFT数据，或者模型经过RLHF，输出的分布与训练集不同），可以通过 自蒸馏 的方式为MEDUSA头训练生成数据集。 使用与原始模型领域相似的公开种子数据集里的提示（例如ShareGPT用于聊天模型），喂给原始模型自己生成回答，形成符合原始模型输出分布的QA对。

对于 MEDUSA-1 来说，直接用该数据集训练MEDUSA 头即可。

对于 MEDUSA-2 ， 仅用自蒸馏数据会降低模型生成质量 ，为了避免主干模型能力退化， 用原始模型的概率分布作为label ，而不是one-hot真实token作为label，主干模型的损失函数为：

【公式开始】L_{\text{LM-distill}} = KL(p^{(0)}_{\text{original}, t} \| p^{(0)}_{t}) 【公式结束】

其中 【公式开始】p^{(0)}_{\text{original},t} 【公式结束】 表示原始模型在位置t的预测概率分布。为了缓解内存压力，采用LoRA微调来训练主干模型，原始模型则是关闭LoRA的主干模型，而无需采用量化。

## 6.7 推理链压缩

思维链是什么？

用户问题：小红有10块钱，她坐地铁花了5块，妈妈又给了她十块，问现在她有多少钱？

模型回复：<think>小红有10块钱，她花了5块钱做地铁，现在还剩5块，妈妈又给了她10块，她有15块现在，所以答案是15。</think>\n<answer>15</answer>

<think></think>之间的内容就是推理链。

为什么需要推理链？

将复杂的问题分解为简单的子任务，提升大模型解决复杂问题的能力。

推理链为什么会延缓模型回复速度？

由于LLM decode 的自回归性质，推理延迟与生成序列长度成正相关。为了提升模型回复速度（TTFT），避免用户长时间等待，就需要对推理链进行压缩，而压缩推理链又会不可避免的影响模型生成质量。如何平衡推理速度和回复质量，就成为了研究的焦点。

过度思考 ，人类社会俗称“ 你想多了 ”，小到你身上不知道从在哪沾了一根长头发让npy彻夜难免，大到做决策时面临大量信息和众多选择难以做出判断，每个人或许都面临过过度思考的场景。而在大模型领域，最经典的案例就是面对“9.8 与 9.11 哪个更大”的问题时，大模型洋洋洒洒、旁征博引的分析了一大堆，然后给出错误的结论。越来越多的例子证明， 更长的思维链并不能保证更高的输出质量 ，并且过度思考引入的计算开销也是显著的。 高效的推理应该是在保持推理能力的同时，尽量压缩推理长度，以提升响应速度和压缩计算成本。

CoT推理 ： 在给出最终答案之前，模型会先生成一系列中间推理步骤，以提升回复的准确率 。CoT的变体包括：

Self‑consistency ：采样多条多样化思考链后，对结果进行边缘化和聚合以选出最一致答案。

Tree‑of‑Thought ：将思考过程组织成一棵可回溯的树，支持并行子任务求解。

Graph‑of‑Thought ：以图结构组织思考节点，支持循环的迭代优化单步思考。

过度思考 ：在CoT思考中， 大模型生成了过于详细或不必要的推理步骤，导致输出过于冗长，严重影响解决问题的效率 。下图展示了询问 Qwen3-235B 和 DeepSeek-R1 “ 2+3等于几 ”时，两个模型都出现了过度思考问题，推理序列过于冗长，增加了推理成本和延迟。

如何制定RL的长度奖励呢？

Demystifying：证明了 CoT长度与准确率不呈正相关关系 ，为了稳定模型性能，应该控制CoT长度，提出了 基于余弦函数的长度奖励设计，对正确回答按余弦曲线平滑奖励；对超出最大长度设定的生成施加“超长惩罚”

第一个问题是，如何收集包含长度不同的CoT推理数据，特别是比较短的CoT数据呢？

长 CoT 数据可以通过提示预训练模型收集 ，基于这些长 CoT 数据，可以通过以下方式收集短CoT数据。

后处理（Post‑Reasoning）压缩

利用规则或更强模型（如 GPT‑4）对已生成的长 CoT 进行压缩，删去冗余推理步骤，仅保留关键信息 （例如 C3oT、TokenSkip 等方法）。

在线（During‑Reasoning）压缩

在推理时 通过prompt控制LLM生成短CoT ：

LearnSkip ：通过 人工或随机跳过部分步骤 ，再以“在 n 步内完成”为提示构造数据集进行SFT，让模型学会简洁思考。

Self-Training ： 生成多条推理路径并选择最短者，收集为训练样本 。

Token‑Budget ： 为推理过程设定最优 token 预算，并以二分搜索方式寻找该预算，再按此生成短推理链 。

参数混合（CoT‑Valve） ： 将“非推理” 【公式开始】θ_N 【公式结束】 与“长推理” 【公式开始】θ_L 【公式结束】 LoRA 参数进行混合 【公式开始】α∆θ_N + (1 − α)∆θ_L 【公式结束】 ，通过调整 【公式开始】\alpha 【公式结束】 来控制生成的推理长度。

第二个问题是，怎么通过微调实现高效推理？

标准 SFT ：使用 LoRA 或 全参数微调 ， 在变长 CoT 数据上最小化语言模型的困惑度或DPO损失函数 。

渐进式微调（Progressive Fine‑tuning） ： 微调过程中逐步减少数据的推理步骤（LearnSkip），或通过将“非推理”与“长推理” LoRA 参数进行混合，逐步增加长推理参数的权重，逐步引导模型学会不同长度的思维链（CoT-Value）。

基于推理输出的高效推理

将显性推理步骤压缩到潜在表示中

《 From Explicit CoT to Implicit CoT: Learning to Internalize CoT Step by Step 》这篇论文中提出，通过 在训练过程中逐步移除显性思考步骤（或添加“思考”标记） ，模型即可在潜在空间“记住”这些推理链，保持或提升推理性能。《 Let’s Think Dot by Dot: Hidden Computation in Transformer Language Models 》这篇论文提出 大模型推理性能的提升并非依赖于人类可读的中间推理步骤，而是增加更多token带来的计算能力提升 ，即便用无意义的填充替代CoT步骤也能增强模型性能。通过替代或压缩文本形式的 CoT推理步骤，能提高推理效率并降低生成长度开销。

那么，怎么将显式的推理步骤压缩到潜在空间呢？

![原文图片](assets/c1284c8767c4.png)

方法大体可以分为两类：

在模型内部直接学习潜在思维：

Coconut (Chain of Continuous Thought)： 将模型最后一层隐藏状态视作“Continuous Thought”，以替代离散的token，将隐藏状态作为下一个输入embedding ，通过逐步添加潜在的CoT token，能有效提升准确率的同时减少显示CoT长度。

CODI (Continuous Distillation of Implicit CoT)： 通过 自蒸馏，让模型同时充当师生同时学习显示和隐式CoT，同时在token上对齐隐藏激活对齐， 在潜在空间学习推理，无需显性 CoT token。

CCOT (Compressed CoT with LoRA)： 先用强模型生成完整 CoT，并挑选关键token对应的隐藏态； 用两个 LoRA模块（CCOT 和 DECODE）分别负责预测关键token，和基于这些token进行解码，还原出简洁 CoT 。

Heima： 针对多模态 LLM， 将每一步长文本推理替换为单个“思考 token” ，并以此更新训练数据，实现高效多模态推理。

Token Assorted： 结合 VQ‑VAE， 将部分 CoT 切换成离散的潜在 token，与文本 token 混合训练 ，以紧凑的潜在形式表示某些推理步骤，兼顾效率与可解释性。

Looped Transformer： 通过在潜在空间对同一层 Transformer 重复多次，模拟更深层次的推理迭代，无需增加参数，使模型能够在潜在空间内进行迭代推理过程。

冻结主模型，增加轻量隐思维模块

SoftCoT： 保持原 LLM 权重不动， 训练一个小型辅助网络输出“软思维 token”并映射至冻结LLM的嵌入空间中 ，显著提升推理表现。

以上方法能有效减少显性推理 token 数量并降低响应延迟，并且在潜在空间的操作具有灵活性和可拓展性，缺点则是 降低了可解释性，不利于对关键决策链路的验证，并且引入了额外的训练。

推理过程中动态推理

与以上不同，本节聚焦于无需额外训练即可在推理阶段动态调整思考策略的方法，通过选用不同的标准，如基于奖励的、基于置信度的和基于一致性的选择性推理，来决定思考长度。

选择什么标准来知道推理呢？什么是高效的推理范式？

主要有以下三个方向：

测试时扩展（Test‑Time Scaling）：

Best‑of‑N 采样： 对同一输入生成 N 条候选 CoT ，采用多数投票或奖励模型评估后选出最优输出，提升复杂任务表现。

Beam Search： 将完整的生成过程分为多个步骤，在每一步保留最优 beam 并利用过程奖励模型筛除落后分支 ，细粒度优化中间生成质量。

Monte Carlo Tree Search (MCTS)： 并行探索多条路径，评估后向回传累积得分，最终选择最优分支 ，适合需要回溯与分支的深层推理。

基于奖励的推理优化

Speculative Rejection： 在 Best‑of‑N 解码中动态丢弃低质量分支，仅保留高分序列以节省算力 。

Reward‑Guided Speculative Decoding (RSD)： 利用草稿模型快速生成候选，使用过程奖励模型动态评估草稿模型的输出，高奖励得分的输出直接接受，低奖励的再由目标模型进一步优化 。

基于置信度的推理优化 ：

Dynamic Parallel Tree Search (DPTS)： 对 Tree‑of‑Thought 进行并行化扩展， 依据置信度提早剪枝，减少冗余探索，平衡 exploration & exploitation。

Certaindex ： 提出一种 基于语义熵和奖励模型得分 衡量LLM推理过程置信度的指标，对高置信度的查询提前终止推理，因为进一步的推理不太可能改变最终答案。

FastMCTS： 基于 MCTS 树， 优先扩展高置信度的轨迹 ，根据问题的复杂性调整树的扩展，提高了效率和推理的多样性。

Length‑filtered Vote： 对不同长度的 CoT 分组投票 ，剔除过短或过长的答案，追求“最优长度”下的高置信回答。

基于一致性标准的推理优化 ：

Self‑Truncation Best‑of‑N (ST‑BoN)： 在 BoN 采样的基础上， 利用 嵌入一致性 判断早期生成序列的重要性，提前终止不一致路径，只保留与多数最接近的样本 ，降低评估成本。

基于摘要的推理优化：

LightThinker： 训练模型学会何时压缩中间思路， 将冗长 CoT 压缩为gist tokens，并通过稀疏注意力掩码聚焦关键摘要 ，兼顾性能与效率。

InftyThink： 实现“无限”迭代推理： 交替生成“思考→摘要”，丢弃旧思路，仅保留最新摘要 ，突破上下文窗口限制并保持高准确度。

基于输入prompt的高效推理

提示词引导

遥想当年，一句“Let the step by step”开启了多少大模型开始了推理之路，当然也可以通过在 prompt 中直接加入长度或表达约束控制大模型推理长度。

那么，具体怎么通过提示词控制LLM的推理长度呢？

Token‑Budget ：通过 设置token预算减少不必要的推理token ，引入 TALE‑EP  ，首先通过提示 LLM 本身来估算合理的 token 预算，然后将预算嵌入到提示词中，引导模型生成更高效的回复。

推理路由

根据用户提示的 难度 或 不确定性 自动“路由”到不同能力的模型或不同推理范式，以兼顾效率和准确。

Sketch‑of‑Thought (SoT) ：用 DistilBERT 做路由器，选择三种思考范式（Conceptual Chaining口头表达、Chunked Symbolism数学符号表达、Expert Lexicons专家表达）；

不确定性

Self‑Ref ：在 LLM 内部微调 不确定度打分专用 token，仅对高不确定性输出触发路由；

Confident or Seek Stronger ：构造校准数据，让模型能在不访问用户输入的前提下预测自身信心，用于 routing 决策。

如何用更少的数据训练推理模型？

最小但高影响力数据筛选

LIMO ： 仅用 817 个精心筛选的示例 （按 难度 、 通用性 、 多样性 挑选问题，按 最优结构组织 、 有效认知 、 验证质量 挑选答案），即可超越此前 10 万+ 示例训练的模型。

s1 ：构建了包含 1,000 个高质量问题–推理对的 s1K 数据集，并 在推理阶段引入“预算强制”（budget forcing），在推理过程中调节推理时间。

自我验证作为信号

S2R ： 先用小量示例做SFT，让模型学会自我验证与自我纠错；再通过 RL 在过程层面与结果层面强化这一能力 。

如何提升小模型（SLM）的推理能力？

知识蒸馏（Distillation）

混合蒸馏 ：结合 长/短 CoT 示例或 CoT + 程序化思路（PoT） ，提升蒸馏效果。

反事实蒸馏 ：通过 遮蔽问题中因果特征来扩充训练集，促使LLM完成被屏蔽的文本 。为每个数据生成正负样本，从多视角蒸馏。

反馈驱动蒸馏 ：从现有问题中创建多样和复杂的问题来迭代扩充蒸馏集，生成多样化、更复杂问题用于训练小模型。

探测与检索蒸馏： 训练两个互补小模型， 一个“探测模型”检索相关知识，另一个“推理模型”据此生成推理链 ，分工协作提升推理质量。

自适应思考蒸馏 ： 让蒸馏过程中的小模型， 基于任务复杂度动态调整推理策略 ，提升对长短不同问题的适应能力。

符号知识内化： 将符号化知识注入 SLM ，帮助其在 CoT 推理中引入结构化、可解释的符号操作，提升效率与准确度。

自我纠错管道： 生成小模型的自我纠错数据，并据此微调，使其具备 先推理→再校正 的能力 ，有效减少逻辑漏洞。

模型压缩

量化 ： 将权重降低到低精度 ，能在大幅减小模型参数量和加速推理的同时，几乎无损保留推理性能；

剪枝 ：移除低重要性权重或神经元，会严重损害多步逻辑能力，显示出剪枝对推理链路的破坏性。

参考论文：《Stop Overthinking: A Survey on Efficient Reasoning for Large Language Models》

### 6.7.2 LightThinker

论文简介

提出一种 在推理过程中动态压缩中间思维链 的方法LightThinker ，将冗长的思维步骤压缩成紧凑的表示。为了实现这一目标，通过 数据构建 、 将隐藏状态映射到紧凑的token 以及 创建专门的注意力掩码 来训练 模型何时以及如何执行压缩实现 。为了量化压缩程度，引入了 Dependency（Dep） 来测量生成过程中对历史token的依赖程度。

原论文： https://arxiv.org/pdf/2502.15589v1

简介

推理模型在 CoT 的基础上，还引入了多种推理能力，例如 试错 、 反思 、 纠错 和 回溯 ，显著提升了解决复杂问题的成功率。但是这 消耗了海量token ，同时使得“快速思考”变成了“慢速思考”，更为严重的是， Transformer的注意力计算复杂度随着上下文长度而二次方增长，KV cache的缓存压力随上下文长度线性增长，严重限制了LLM在长文本生成和复杂推理任务上的效率。

Qwen2.5-32的上下文达到 【公式开始】10^4 【公式结束】 时，KV cache占用的空间和模型本身相当。

现有方法主要分为两类，详细可参考 Stop Overthinking

通过 提示词工程 或者 专门训练 ，引导大模型在推理过程中生成更少的token，需要更复杂的数据构建和迭代优化。

推理过程中动态减少推理步骤和输出长度 ，选择性保留KV cache中的重要部分，推理过程中需要对token的重要性进行评估，导致推理延迟增加。

LightThinker

Denotation：小写字母 【公式开始】x_i 【公式结束】 表示当个token；大写字母 【公式开始】X 【公式结束】 表示token序列； 【公式开始】[\cdot] 【公式结束】 表示特殊token， 【公式开始】<\cdot> 【公式结束】 表示可选特殊token。

LightThinker主要解决了两个关键问题：

何时压缩？ token级压缩 在生成固定数量的token后进行压缩 ，可能忽略语义边界； thought级压缩 在生成一个完整的“思维”（如一个句子或段落）后进行压缩 ，能够更好地保留语义信息，但需要更复杂的分割函数。LightThinker通过重构数据解决这个问题。

如何压缩？ 文本压缩 将当前thought压缩为为较短的文本或连续向量块 ，需要额外的编码模型并增加计算开销； 隐藏状态压缩 则将当前thought的隐藏状态压缩为少量特殊token（gist token）的隐藏状态 ，不需要额外的模型。LightThinker采用后者，构建基于thought的注意力掩码。

数据重建

给定输入X和输出Y，使用分割函数 【公式开始】Seg(\cdot) 【公式结束】 将Y划分为 【公式开始】k 【公式结束】 个子序列 【公式开始】S 【公式结束】 ， 【公式开始】Y = \{S_i\}_{i=1}^k 【公式结束】 ， 可以进行token级和thought级的分割 。

Seg的实现

token级：每6个token压缩为两个token，即 【公式开始】|C| = 2 【公式结束】 .

实验

实验设置

使用Bespoke-Stratos-17k数据集（B17K），在DeepSeek-R1-Distill-Qwen-7B和Llama3.1-8B上进行全参数指令微调，模型记为Vanilla。

对比模型：H2O（采用贪心，保留历史token中最高累积注意力的token）和SepLLM（保留标点处的token）是无需训练的加速方法，AnLLM是一种基于训练的压缩方法。

评测集：GSM8K、MMLU、GPQA、BBH。

评估指标：准确率和效率，包括推理时间、峰值令牌数Peak、Dep（每个生成令牌对历史令牌的依赖总和，值越低表示压缩效果越好）。Peak表示某一时刻的状态，Dep表示整个推理过程。

训练细节：采用贪心解码，最大输出长度10240个token。

![原文图片](assets/faa461c3dbff.jpeg)

为什么要设计Dep？

LightThinker和An-LLM这种动态压缩方法， 推理过程中的压缩次数和压缩比都由LLM决定 ，而H2O和SepLLM这些无需训练的方法，可以通过设置超参数控制推理过程中保留的最大token量，这导致无法用峰值KV cache缓存来衡量。如下图所示，尽管LightThinker的峰值KV cache高于H2O，但峰值只在瞬间出现，而H2O则长时间保持较高token数目。

此外，H2O只压缩prompt，并假设prompt长度固定，从而预定义压缩比。 而LightThinker对输出也进行压缩，无法预设压缩比。

Dep的定义

Dep衡量生成过程中对历史token的依赖总量 ，在确保有效压缩比相似的情况下进行评估，即对应下图中曲线的面积。 Dep值越低表示压缩越有效 。

Vanilla：面积为直角梯形，即随着生成依赖越来越大

H2O：面积为梯形+矩形，即生成依赖先增大到阈值（达到最大token数）后不变。

LightThinker：在生成过程中实时计算，没有显式解。

![原文图片](assets/1418bb4c84cf.png)

主要结果

在B17K上训练后，Vanilla模型显著优于CoT和Distill-R1。

对比Vanilla，实验结果表明，Lightthinker实现了有效性和推理效率之间的平衡：

在Qwen上，牺牲1%的准确性，就节省了26%推理时间，减少了70%峰值token使用量和78的Dep。

在Llama上，牺牲6%的准确性，就节省了1%推理时间，减少了70%峰值token使用量和74%的Dep。

Seg的实现中， thought级显著优于token级，因为token级的分割会导致语义边界的丢失。
